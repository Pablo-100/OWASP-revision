3) SOP (Same-Origin Policy)

Regle navigateur: un script d un site ne lit pas facilement les donnees d un autre site, sauf si protocole + hote + port sont identiques.

http://a.com/x + http://a.com/y = OK
http://a.com + https://a.com = KO (protocole)
http://a.com + http://api.a.com = KO (hote)
http://a.com + http://a.com:8080 = KO (port)

Chapitre 2

Top 10 OWASP 2021 en mode ultra clair

A01 Broken Access Control

Scene: tu changes un id URL et tu vois les donnees d un autre user.

Fix: verifier droits cote serveur sur chaque action.

A02 Cryptographic Failures

Scene: mot de passe mal stocke, trafic non protege.

Fix: TLS fort, hash sale fort, gestion cle stricte.

A03 Injection (SQL/XSS/Command)

Scene: entree user executee comme commande.

Fix: requetes preparees, validation allowlist, encoding sortie.

A04 Insecure Design

Scene: process metier faible des la conception.

Fix: threat modeling + abuse cases avant dev.

A05 Security Misconfiguration

Scene: comptes par defaut, debug actif, directory listing.

Fix: hardening + secure defaults + secrets propres.

A06 Vulnerable Components

Scene: dependance obsolete exploitee (ex Log4j style).

Fix: inventaire + scan + patch rapide.

A07 Auth Failures

Scene: session fixation, brute force, mots de passe faibles.

Fix: MFA, rate limit, timeout, regen session id.

A08 Software/Data Integrity Failures

Scene: update non signee ou source non fiable.

Fix: signatures, pipeline CI/CD securisee.

A09 Logging & Monitoring Failures

Scene: attaque detectee apres des mois.

Fix: logs utiles, SIEM, alerting, plan incident.

A10 SSRF

Scene: ton serveur appelle un endpoint interne dangereux.

Fix: URL allowlist + blocage egress interne + validation DNS/IP.

Descriptions avancees

Comprendre "kl chy": pourquoi, comment, impact, correction

A01-A03 (Acces, Crypto, Injection)
  • A01: la logique d autorisation doit etre verifiee sur chaque endpoint, pas juste en UI.
  • A02: la crypto faible transforme une fuite en compromission massive.
  • A03: toute entree non maitrisee peut devenir commande interpretee.
  • Pattern defense: deny-by-default, prepared statements, secrets management.
A04-A07 (Design, Config, Components, Auth)
  • A04: un mauvais design n est pas patchable facilement sans refonte.
  • A05: les defaults non modifies sont une porte directe.
  • A06: dependance obsolete = faille heritee, meme si ton code est propre.
  • A07: identite + session mal gerees = account takeover.
A08-A10 (Integrite, Monitoring, SSRF)
  • A08: sans verification integrite, la chaine de livraison logiciel devient vecteur d attaque.
  • A09: si tu ne vois rien dans les logs, l attaquant peut rester longtemps.
  • A10: SSRF permet de pivoter vers reseau interne et metadata cloud.
  • Pattern defense: signatures, SIEM/alerting, egress control, URL allowlist.
